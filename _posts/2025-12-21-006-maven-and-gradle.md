## 1. Maven과 Gradle은 무엇인가?

Maven과 Gradle은 Java 진영에서 가장 널리 사용되는 **빌드 관리 도구(Build Tool)**입니다.  
Spring Boot 프로젝트를 생성할 때도 대부분 이 둘 중 하나를 선택하게 됩니다.

두 도구는 동일한 목적을 가지고 있지만,  
접근 방식과 설계 철학에는 분명한 차이가 있습니다.

이를 이해하기 위해 먼저 **빌드란 무엇인지**부터 살펴보겠습니다.

---

## 2. 빌드란 무엇인가?

빌드는 단순히 소스 코드를 컴파일하는 작업만을 의미하지 않습니다.

일반적으로 빌드는 다음 과정을 포함합니다.

- 소스 코드 컴파일
- 의존성 라이브러리 다운로드
- 테스트 코드 실행
- 실행 가능한 결과물(JAR/WAR) 생성
- 배포를 위한 패키징

즉, 개발자가 작성한 코드가  
**실제로 실행 가능한 애플리케이션으로 변환되는 전체 과정**을 빌드라고 볼 수 있습니다.

---

## 3. 빌드 관리 도구란?

빌드 관리 도구는 이러한 빌드 과정을 **자동화하고 표준화**해주는 도구입니다.

대표적인 역할은 다음과 같습니다.

- 의존성 관리
- 빌드 생명주기 관리
- 반복 작업 자동화
- 플러그인 기반 확장

초기 Java 프로젝트에서는 Ant와 같은 도구를 사용해
개발자가 직접 빌드 스크립트를 관리했지만,
표준이 없어 프로젝트마다 구조가 달라지는 문제가 있었습니다.

이 문제를 해결하기 위해 등장한 도구가 **Maven**입니다.

---

## 4. Maven의 정의와 특징

### 4-1. Maven의 등장 배경

Maven은 Ant의 단점을 보완하기 위해 등장했습니다.

- 프로젝트마다 제각각이던 구조
- 복잡한 의존성 관리
- 협업 시 발생하는 설정 차이

이를 해결하기 위해 Maven은  
**정해진 규칙과 표준**을 중심으로 설계되었습니다.

---

### 4-2. Maven의 정의

Maven은 **Convention over Configuration(설정보다 관례)** 철학을 기반으로 한 빌드 관리 도구입니다.

정해진 디렉토리 구조와 빌드 생명주기를 따르기만 하면,
비교적 적은 설정으로도 프로젝트를 일관되게 관리할 수 있습니다.

---

### 4-3. Maven의 특징

- XML 기반 설정 파일 (`pom.xml`)
- 표준 디렉토리 구조
- 명확한 빌드 생명주기
- 안정성과 예측 가능성

Spring Boot 공식 문서와 예제 대부분이
오랫동안 Maven 기준으로 제공되어 왔기 때문에,
처음 접하는 사람에게는 진입 장벽이 낮다는 장점도 있습니다.

---

## 5. Gradle의 정의와 특징

### 5-1. Gradle의 등장 배경

Maven은 표준화된 구조라는 강점을 가졌지만,
설정이 복잡해질수록 다음과 같은 한계도 드러났습니다.

- XML 설정의 가독성 문제
- 조건 분기나 복잡한 빌드 로직 표현의 한계
- 대규모 프로젝트에서의 빌드 성능

Gradle은 이러한 문제를 해결하기 위해 등장했습니다.

---

### 5-2. Gradle의 정의

Gradle은 **유연성과 성능을 개선한 빌드 관리 도구**입니다.

단순한 설정 파일이 아니라,
빌드 자체를 하나의 **프로그램처럼 작성**할 수 있도록 설계되었습니다.

---

### 5-3. Gradle의 특징

- Groovy 기반 DSL (`build.gradle`)
- 코드 형태의 빌드 스크립트
- Incremental Build, Cache 기반의 빠른 빌드
- 멀티 모듈 프로젝트에 강점

최근 Spring Boot 프로젝트에서는
Gradle을 기본 선택지로 사용하는 경우도 점점 늘어나고 있습니다.

---

## 6. Groovy란 무엇인가?

Gradle을 이해하기 위해 Groovy를 간단히 살펴볼 필요가 있습니다.

Groovy는 **JVM 위에서 동작하는 동적 언어**로,
Java와 100% 호환됩니다.

- Java보다 간결한 문법
- 동적 타입 지원
- DSL 작성에 적합

Gradle은 이러한 Groovy의 특성을 활용해
빌드 설정을 단순한 설정 파일이 아닌
**프로그래밍 가능한 스크립트**로 확장했습니다.

---

## 7. Maven과 Gradle 한눈에 비교

| 항목 | Maven | Gradle |
|------|-------|--------|
| 설정 방식 | XML (`pom.xml`) | Groovy / Kotlin DSL |
| 철학 | Convention over Configuration | Flexibility & Performance |
| 가독성 | 설정이 길어질수록 낮아짐 | 코드 기반으로 상대적으로 높음 |
| 유연성 | 낮음 | 매우 높음 |
| 학습 난이도 | 낮음 | 상대적으로 높음 |
| 빌드 속도 | 보통 | 빠름 |
| 멀티 모듈 | 설정 복잡 | 비교적 간단 |

---

## 8. Maven과 Gradle에 대한 의견이 갈리는 이유

Maven과 Gradle에 대한 평가는
단순한 성능 비교보다는
**개발자가 어떤 환경과 프로젝트를 경험했는지**에 따라 크게 달라집니다.

---

### 8-1. Maven이 더 좋다고 느끼는 이유

- 정형화된 구조로 협업이 쉽다
- 설정이 명확하고 예측 가능하다
- 엔터프라이즈 및 레거시 환경에서 검증되었다

특히 CI/CD 환경이나
여러 팀이 함께 사용하는 공통 프로젝트에서는
이러한 안정성이 장점으로 언급되는 경우가 많습니다.

---

### 8-2. Gradle이 더 좋다고 느끼는 이유

- XML보다 코드 기반 설정이 직관적이다
- 복잡한 빌드 로직을 자연스럽게 표현할 수 있다
- 멀티 모듈 구성과 확장에 유리하다
- 대규모 프로젝트에서 빌드 성능이 좋다

Spring Boot에서 환경별 설정이나
멀티 모듈 구조를 다뤄본 경우,
Gradle의 유연함이 장점으로 평가되는 경우도 많습니다.

---

### 8-3. “Gradle로 하면 깨진다”는 말이 나오는 이유

이번에 Maven과 Gradle을 비교하며 정리하는 과정에서,
"Gradle로 하면 가끔 깨진다"는 표현을 종종 찾아볼 수 있었습니다.

여기서 말하는 ‘깨진다’는 것은
애플리케이션 자체의 문제가 아니라,
**빌드 과정에서 예상치 못한 오류를 경험하는 경우**를 의미하는 경우가 많았습니다.

Gradle은 자유도가 높은 만큼
Gradle 버전, Spring Boot 플러그인 버전,
각종 서드파티 플러그인 간의 조합에 따라
빌드 결과가 달라질 수 있습니다.

실제로 Gradle을 사용하면서
Gradle 버전이나 플러그인 버전을 변경한 이후
빌드가 실패하거나,
기존에는 없던 오류가 발생한 경험도 있었습니다.

또한 캐시와 Incremental Build 기능은
빌드 속도를 크게 개선해주지만,
캐시로 인해 원인을 파악하기 어려운 오류를 만들기도 합니다.

이러한 경험들이 누적되면
Gradle이 상대적으로 불안정하다는 인식으로 이어질 수 있겠다는 생각이 들었습니다.

반면 Maven은 정형화된 구조와 제한된 설정 덕분에
예측 가능한 빌드 결과를 제공하고,
이 점이 “안정적이다”라는 평가로 이어지는 경우가 많아 보였습니다.

---

### 8-4. Spring Boot 기준으로 느낀 개인적인 경험

Spring Boot 프로젝트를 진행하면서,
개인적으로는 Maven보다 Gradle이 더 직관적이고 편리하게 느껴졌습니다.

Maven을 사용할 때는 `pom.xml`이 길어질수록
전체 빌드 흐름을 파악하기 어려웠고,
환경별 설정이나 조건 처리를 추가할 때 부담이 있었습니다.

반면 Gradle에서는
빌드 설정이 코드로 드러나 있어
현재 어떤 설정이 적용되는지 이해하기 쉬웠고,
멀티 모듈 구성이나 확장도 상대적으로 수월했습니다.

물론 Gradle 역시
버전 관리나 설정을 제대로 하지 않으면
빌드 오류를 겪을 수 있다는 점을 경험했지만,
그만큼 자유도가 높은 도구라는 의미로도 느껴졌습니다.

이러한 이유로 개인 프로젝트나
새롭게 구성하는 Spring Boot 환경에서는
Maven보다 Gradle을 더 선호하게 되었습니다.

## 10. 마무리

Maven과 Gradle은 모두 훌륭한 빌드 관리 도구이며,
각각의 철학과 장단점이 분명합니다.

중요한 것은 어떤 도구가 더 낫냐가 아니라,
**현재 프로젝트와 팀에 어떤 도구가 더 잘 맞는지**라고 생각합니다.

개인적으로는 Spring Boot 기준에서
Gradle이 더 직관적이고 유연하게 느껴졌지만,
안정성과 표준이 중요한 환경에서는
여전히 Maven이 좋은 선택이 될 수 있다고 느껴졌습니다.

앞으로는 도구 자체보다, 왜 이런 선택지가 존재하는지를 이해하고
각 상황에 맞는 선택을 고민하는 과정이 더 중요하다고 생각합니다.
