## 1. CI/CD란 무엇일까?

CI/CD는  
**지속적 통합(Continuous Integration)** 과  
**지속적 배포(Continuous Deployment)** 를 의미합니다.

간단히 말해,  
코드를 저장소에 반영하면  
빌드 → 테스트 → 배포 과정이 자동으로 이어지도록 만드는 흐름입니다.

CI/CD를 도입하면 다음과 같은 효과를 기대할 수 있습니다.
 
- 배포 과정이 사람의 기억이나 감각에 의존하지 않음
- 반복적인 빌드·배포 작업을 자동화
- 배포 실수와 환경 차이로 인한 오류 감소
- 작은 변경 사항도 부담 없이 빠르게 반영 가능

이제 CI/CD는 대규모 서비스뿐만 아니라  
규모가 크지 않은 프로젝트에서도  
점점 기본적인 개발 환경으로 자리 잡고 있습니다.

---

## 2. 대표적인 CI/CD 도구

CI/CD를 구현하는 방법은 다양하며,  
프로젝트의 규모나 환경에 따라 선택지는 달라질 수 있습니다.

대표적으로 다음과 같은 도구들이 자주 언급됩니다.

- **Jenkins**  
  가장 오래된 CI/CD 도구 중 하나로,  
  직접 서버를 운영하며 높은 자유도의 파이프라인 구성이 가능합니다.

- **GitHub Actions**  
  GitHub에 기본으로 제공되는 CI/CD 기능으로,  
  별도의 서버 없이 빠르게 CI/CD를 도입할 수 있습니다.

- **GitLab CI**  
  GitLab 환경에 최적화된 CI/CD 도구로,  
  GitLab을 사용하는 팀에서 자연스럽게 선택되는 경우가 많습니다.

- **CircleCI 등 기타 SaaS CI/CD 도구**  
  설정 편의성과 빠른 도입이 장점인 서비스형 CI/CD 도구들입니다.

CI/CD 도구는 이처럼 선택지가 다양하고,  
각각의 장단점과 지향점도 서로 다릅니다.

---

## 3. CI/CD 도구를 고를 때 마주하게 되는 고민

CI/CD를 도입하려고 하면  
가장 먼저 마주하게 되는 질문은 보통 다음과 같습니다.

> “CI/CD면 Jenkins 아닌가?”

실제로 Jenkins는 오랫동안 사실상의 표준처럼 사용되어 왔고,  
지금도 많은 기업과 프로젝트에서  
핵심 CI/CD 도구로 활용되고 있습니다.

다만 프로젝트를 하나씩 경험하다 보니,  
**모든 상황에서 Jenkins가 반드시 필요한 선택일까?**  
라는 의문도 함께 들기 시작했습니다.

특히 규모가 크지 않은 프로젝트나  
복잡한 배포 전략이 필요하지 않은 경우라면,  
다른 선택지도 충분히 검토해볼 만하다고 느꼈습니다.

---

## 4. Jenkins와 GitHub Actions의 기본적인 성격 차이

이번 포스팅에서는  
대표적인 CI/CD 도구로 자주 비교되는 다음 두 가지만 살펴보았습니다.

- **Jenkins**
- **GitHub Actions**

두 도구는 CI/CD라는 같은 목적을 가지고 있지만,  
접근 방식과 철학은 꽤 다릅니다.

---

## 5. Jenkins는 어떤 경우에 선택할까?

Jenkins는 전통적인 CI/CD 자동화 서버입니다.

### Jenkins의 특징
- 별도의 Jenkins 서버를 직접 구축해 운영
- 플러그인 기반으로 거의 모든 CI/CD 시나리오 구현 가능
- 복잡한 파이프라인과 사내 인프라 연동에 강점

### Jenkins가 잘 맞는 경우
- 빌드·배포 단계가 매우 복잡한 경우
- 사내 시스템이나 레거시 환경과 긴밀한 연동이 필요한 경우
- CI/CD 자체를 세밀하게 제어해야 하는 경우

Jenkins는 범용성이 높은 도구이지만,
그만큼 **운영하고 관리해야 할 시스템**이기도 합니다.

---

## 6. GitHub Actions는 어떤 성격의 도구일까?

GitHub Actions는  
GitHub에 기본으로 제공되는 CI/CD 기능입니다.

### GitHub Actions의 특징
- `.github/workflows` 디렉터리에 YAML 파일로 워크플로우 정의
- push, pull request 등의 이벤트 기반 실행
- 별도의 CI 서버 구축 없이 바로 사용 가능

### GitHub Actions가 잘 맞는 경우
- 프로젝트 규모가 크지 않은 경우
- 빌드 및 테스트 시간이 길지 않은 경우
- 단순한 빌드 → 테스트 → 배포 흐름
- CI/CD를 빠르게 도입하고 싶은 경우

이런 조건이라면, **굳이 Jenkins가 아니어도
GitHub Actions만으로 충분히 목적을 달성할 수 있다**고 생각합니다.

---

## 7. Jenkins vs GitHub Actions 간단 비교

| 항목 | Jenkins | GitHub Actions |
|----|--------|---------------|
| CI 서버 | 직접 구축 필요 | GitHub 제공 |
| 설정 난이도 | 높음 | 낮음 |
| 파이프라인 자유도 | 매우 높음 | 중간 |
| 운영 부담 | 큼 | 거의 없음 |
| 단순 CI/CD | 과함 | 적합 |
| 복잡한 CI/CD | 매우 적합 | 제한적 |

이 비교에서 중요한 점은  
**어느 쪽이 더 우수한가가 아니라,  
어떤 상황에 더 적합한가**라고 생각합니다.

---

## 8. GitHub Actions의 비용 구조와 최근 이슈

GitHub Actions는 “무료 CI/CD 도구”로 인식되지만,  
정확히 말하면  
**저장소 유형과 사용량에 따라 비용 구조가 달라지는 서비스**입니다.

- Public repository: 무료
- Private repository: 사용량 기반 과금 구조

공부용 프로젝트에서는 큰 문제가 없을 수 있지만,  
실제 서비스용 프로젝트는 보안과 운영상의 이유로  
대부분 private repository를 사용합니다.

따라서 GitHub Actions 역시  
사용량에 따라 비용이 발생할 수 있다는 점을  
염두에 둘 필요가 있습니다.

### 요금 정책 변경 관련 주요 내용

2024년 12월, GitHub는  
GitHub Actions의 **2026년 요금 정책 변경안**을 발표했습니다.

- [GitHub Actions 2026년 요금 정책 변경안](https://resources.github.com/actions/2026-pricing-changes-for-github-actions/)

주요 내용은 다음과 같습니다.

- GitHub 호스티드 러너 요금 인하 계획
- 셀프 호스티드 러너에 대한 플랫폼 사용 요금 부과 예고
- 이후 사용자 피드백을 반영해 정책 일부 조정

### 이번 요금 변경은 ‘인상’일까, ‘인하’일까?

이번 변경은  
단순한 인상이나 인하로 보기보다는,  
**러너 방식에 따라 영향이 달라지는 구조**로 이해하는 것이 더 적절합니다.

- GitHub 호스티드 러너  
  → 요금 인하로 인해 비용 부담 감소 가능성

- 셀프 호스티드 러너  
  → 기존에는 없던 플랫폼 사용 요금이 추가될 예정

### 셀프 호스티드 러너 과금의 의미

셀프 호스티드 러너는  
사용자가 직접 준비한 서버에서  
GitHub Actions 작업을 실행하는 방식입니다.

과금 대상은 서버 자체가 아니라,  
**작업을 관리·조율·감시하는 GitHub Actions 플랫폼 사용**입니다.

이는 GitHub Actions가  
단순한 기능을 넘어  
**대규모 CI/CD 플랫폼으로 운영되고 있음을 보여주는 변화**라고 볼 수 있습니다.

## 9. 마무리

Jenkins와 GitHub Actions 중  
어느 도구가 더 뛰어나다고 단정할 수는 없다고 생각합니다.

중요한 것은  
**프로젝트의 규모, 단계, 그리고 지금 필요한 수준의 자동화**입니다.

단순한 CI/CD 요구사항이라면 굳이 Jenkins를 사용하지 않더라도  
GitHub Actions를 선택하는 것 역시 충분히 메리트 있는 선택이 될 수 있습니다.

다만 GitHub Actions 역시  
장기적으로는 비용 구조 변화가 있을 수 있기 때문에,  
도입 전에는 항상 최신 정책을 한 번 더 확인하는 것이 필요하다고 생각합니다.
